# 壹零贰肆老友记（IterLife Reunion）MVP 后续产品开发计划

最后更新：2026-03-17  
适用范围：`iterlife-reunion`（后端）+ 对应前端联动能力  
评估依据：`/docs` 既有文档 + 当前仓库已实现代码能力

## 1. 当前 MVP 盘点（基于已实现能力）

| 模块 | 目标 | 当前完成度 | 现状摘要 |
|---|---|---:|---|
| GitOps 内容同步 | GitHub Push 自动同步文章元数据 | 88% | Webhook 签名校验、异步同步、compare fallback、FrontMatter 规范化与重复投影跳过已具备 |
| 文章接口 | 列表/详情/标签/搜索 | 82% | RESTful 路由完备，支持 query 形式多标签筛选，详情可回源 GitHub 拉取 Markdown |
| 搜索能力 | Meilisearch 优先 + DB 兜底 | 70% | 已接入 Meili 开关与查询，缺高亮、召回调优、重建运维工具 |
| 用户体系 | GitHub OAuth + 邮箱绑定 + JWT | 75% | 登录闭环基本可用，安全策略与风控仍需补齐 |
| 评论系统 | 两级评论（主评+回复） | 72% | 发布与读取已完成，两级结构规则已限制 |
| 数据模型 | 读模型 + FrontMatter 规范化记录 + 软删除 | 85% | 当前核心表为 `article` / `article_tag` / `article_comment` / `sys_user` / `article_frontmatter` |
| 部署与安全入口 | GHCR 镜像 + webhook 部署 + HTTPS | 90% | 当前标准流程已切到 GitHub Actions 推镜像、统一 webhook 回调部署 |
| 可观测性与质量 | 监控、测试、CI | 30% | 自动化测试与 CI 覆盖不足，线上质量门槛偏弱 |
| 前端体验增强 | Mermaid/Shiki、项目看板、社交链路 | 25% | 在 PRD/SDR 有定义，代码落地仍不足 |

## 2. 开发优先级矩阵（建议执行顺序）

优先级定义：`P0` 立即影响可用性与验收；`P1` 影响体验与增长；`P2` 影响长期效率与品牌力。

| 工作项 | 优先级 | 重要性 | 当前完成度 | 技术难点 | 高投入点（人日） | 建议结果 |
|---|---|---|---:|---|---:|---|
| `/articles` 数据链路稳定性与可恢复 | P0 | 高 | 75% | Webhook 事件一致性、重复投递、分支过滤 | 2-4 | “Failed to load articles.” 场景归零，支持自愈重刷 |
| Webhook 幂等与补偿机制 | P0 | 高 | 60% | 事件去重、失败重试、死信处理 | 3-5 | 同步失败可追踪、可重放 |
| 接口限流与防刷上线 | P0 | 高 | 20% | 基础设施选型、用户/IP 双维限流、误杀控制 | 2-3 | 登录/评论/Webhook 防刷达标 |
| 统一身份与权限体系（AuthN/AuthZ + 用户模型迁移） | P0 | 高 | 40% | 会话模型、RBAC 权限模型、`sys_user`/`reunion_user` 兼容迁移、跨子站互认 | 5-10 | 认证管会话，授权管权限，一次登录多子站互通，减少用户主数据冗余 |
| 自动化测试基线（单元+集成） | P0 | 高 | 15% | 外部依赖 Mock、测试数据隔离 | 4-7 | 至少覆盖核心链路（同步/认证/评论） |
| 搜索体验增强（高亮/容错/排序） | P1 | 高 | 45% | Meili 索引设计、搜索相关性调优 | 3-5 | 搜索结果“可解释、可用、可扩展” |
| Markdown 渲染增强（Mermaid/Shiki） | P1 | 中高 | 30% | SSR 渲染性能、XSS 安全处理 | 4-6 | 技术内容展示力明显提升 |
| 评论体验升级（Emoji、交互反馈） | P1 | 中 | 60% | 前后端数据结构兼容、敏感内容处理 | 2-4 | 评论参与率提升 |
| `/projects` 开源项目看板落地 | P1 | 中 | 10% | GitHub API 限流、缓存策略 | 3-4 | 首页/子页可形成“作品证明” |
| CI/CD 标准化（GitHub Actions + 发布门禁） | P1 | 高 | 35% | 多仓库联动、回滚策略 | 3-5 | 发布变更更可控、可追溯 |
| 监控与告警（应用/容器/证书） | P2 | 中高 | 25% | 指标采集、阈值设置、告警降噪 | 3-4 | 线上故障发现更及时 |
| 数据备份与恢复演练 | P2 | 中高 | 20% | 备份窗口、恢复校验、演练流程 | 2-3 | 降低不可逆数据风险 |

## 3. 分阶段路线图（8-10 周）

## 阶段 A（第 1-2 周）：MVP 验收稳态化

目标：先把“可用性”打穿，确保对外演示和持续迭代不掉链子。  
范围：
- `/articles` 联调排障与端到端验收脚本
- Webhook 幂等、失败重试、人工重放入口
- 接口限流 / 防刷基础设施落地（当前尚未接入 Redis）
- 统一身份模型落地（认证/授权拆分，先双读单写）
- 基础用例补齐（至少 15-25 条关键测试）

验收标准：
- 文章列表连续 7 天无“加载失败”告警
- Webhook 失败可追踪且可人工重放
- 关键接口 P95 延迟稳定

## 阶段 B（第 3-5 周）：体验可评审化

目标：持续可评审页面 + 核心体验提升。  
范围：
- Markdown 渲染增强（Mermaid/Shiki）
- 评论区体验优化（回复提示、错误反馈、Emoji）
- `/projects` 子页面 MVP（5-7 个仓库卡片 + 社交链接）
- 搜索结果高亮与相关性优化

验收标准：
- 每周可交付可评审页面版本
- 搜索点击率、平均停留时长提升

## 阶段 C（第 6-8 周）：工程化与运营化

目标：把项目从“能跑”升级为“可长期运营”。  
范围：
- CI/CD 门禁（构建、测试、镜像、发布检查）
- 监控告警与证书续期巡检
- 数据备份恢复演练
- 发布回滚 Runbook 固化

验收标准：
- 发布前自动门禁可阻断高风险变更
- 故障恢复流程在演练中可闭环

## 4. 关键技术难点与应对建议

| 难点 | 风险 | 建议方案 |
|---|---|---|
| Webhook 事件一致性 | 丢事件或重复同步导致脏数据 | 增加事件指纹（repo+after+commit）、幂等表、重试队列 |
| Markdown 富渲染安全 | XSS 与 SSR 性能抖动 | 渲染前清洗 + 白名单策略 + 大文档分段渲染 |
| 搜索质量 | 命中率低、结果排序不稳定 | 统一索引字段、引入摘要高亮、按发布时间/热度混排 |
| OAuth 账号安全 | 邮箱绑定冲突、token 滥用 | 完整审计日志、token 过期与刷新策略、异常登录检测 |
| 多仓库发布协同 | 版本错配导致线上异常 | 统一发布版本号、发布清单、回滚对照表 |

## 5. 高投入点（需要提前锁资源）

| 高投入项 | 预计投入 | 原因 |
|---|---:|---|
| 自动化测试体系从 0 到 1 | 4-7 人日 | 需要搭建测试夹具、Mock 外部服务、覆盖关键路径 |
| Markdown 渲染增强（Mermaid/Shiki） | 4-6 人日 | 涉及 SSR、性能、安全三方平衡 |
| CI/CD 全链路门禁 | 3-5 人日 | 多仓库协同与发布流程统一复杂度高 |
| 搜索体验升级 | 3-5 人日 | 相关性调优和索引策略需反复迭代 |

## 6. 管理与执行建议（强建议）

1. 把 P0 项拆成“可在 1 天内验收”的小任务，避免大任务长时间不可见。  
2. 每周固定一次“联调日”，前后端+运维一起跑发布回归清单。  
3. 为 `/articles`、`/api/auth/*`、`/api/comment/*` 建立最小可用 SLO（成功率、延迟、错误率）。  
4. 保持“本地分支 -> PR -> 合并发布分支 -> GitHub Actions 构建 GHCR -> webhook 回调部署”的唯一标准流程，不回退到源码拉取式部署。  
5. 尽快补齐 CI 最低门禁：编译通过 + 核心测试通过 + 镜像可构建。  
6. 建议建立一份统一的“版本发布记录”文档，记录每次发布的 commit、脚本、回滚点。

## 7. 推荐的下一步（本周可执行）

1. 先完成阶段 A 的核心 P0 子项并形成验收记录。  
2. 同步创建阶段 B 的页面原型任务，确保“持续可评审”不断档。  
3. 并行准备阶段 C 的 CI 基础脚手架，避免后期集中返工。
