# IterLife 版本矩阵与版本台账

最后更新：2026-04-11

本文档是 IterLife 体系当前唯一正式版本台账，用于统一记录各业务应用的当前版本、发布状态、标签基线、主要功能和后续版本治理规则。

## 1. 版本模型

IterLife 当前采用两层版本模型：

1. 应用版本
   每个业务仓库独立维护自己的语义化版本，用于代码声明、镜像标识、发布追踪和回滚定位。
2. 系统版本矩阵
   `iterlife-stack` 统一记录某一时点各应用版本组合，作为跨仓协作和发布规划基线。

当前约定：

- Java 服务在 `pom.xml` 中声明版本。
- Node / Nuxt / Vite 前端在 `package.json` 中声明版本。
- 已正式发布的应用使用稳定版本号，例如 `1.1.0`。
- 正在开发、未形成正式发布基线的应用使用 `-SNAPSHOT`，例如 `0.1.0-SNAPSHOT`。
- 正式发布标签统一使用 `release_vX.Y.Z`。
- 历史遗留标签允许保留在台账中说明，但后续版本治理不再延续旧命名规范。

## 3. 当前版本规划

### 3.1 内容系统版本线

`iterlife-reunion` 与 `iterlife-reunion-ui` 当前已进入 `1.1.x` 线，基线含义如下：

- 继承 `v1.0.0` 已完成的文章发布、阅读、分享和个人主页能力。
- 后端补强了文章修改时间刷新等内容投影细节。
- 前端收敛了文章详情页标题与操作区，移除了已下线的 PDF 下载 / 预览路径。
- 当前正式发布标签为 `release_v1.1.0`。

### 3.2 花多少版本线

`iterlife-expenses` / `iterlife-expenses-ui` 当前统一调整到 `1.1.0` 版本线，基线含义如下：

- 后端已在代码内声明正式版本 `1.1.0`。
- 前端已在代码内声明正式版本 `1.1.0`。
- 当前正式发布标签统一为 `release_v1.1.0`。
- 旧的 `openclaw-expenses_v*` 标签已退出当前标准版本线。

### 3.3 身份管理版本线

`iterlife-idaas` / `iterlife-idaas-ui` 当前都处于 `0.1.0-SNAPSHOT` 阶段，说明：

- 已完成仓库与工程骨架建立。
- 尚未形成第一个正式可发布里程碑。
- 在认证、账户、会话和 SSO 最小闭环落地之前，不发布正式 tag。

## 5. 统一治理规则

### 5.1 必须有唯一版本源

- 每个应用仓库必须在代码仓内保留唯一正式版本源。
- Java 仓库使用 `pom.xml`。
- Node 仓库使用 `package.json`。
- Python 仓库若当前缺少正式版本源，应尽快补齐，例如 `pyproject.toml` 或独立 `VERSION` 文件。

### 5.2 正式发布规则

- 正式发布版本必须是稳定语义化版本，不带 `-SNAPSHOT`。
- 正式发布必须创建 `release_vX.Y.Z` 标签。
- 正式发布后，必须同步更新本台账。

### 5.3 开发基线规则

- 尚未具备正式发布条件的服务保持 `-SNAPSHOT`。
- `-SNAPSHOT` 阶段不发布正式 tag。
- 若某仓库还在使用历史标签体系，应先完成版本治理再继续发新标签。

### 5.4 文档同步规则

- 调整任一应用版本号时，必须同步更新本文件。
- 若版本变更对应明确里程碑，建议同步补充对应应用的平铺 release notes，例如 `reunion_release_release_notes_v1_1_0_20260411.md`。
- 版本矩阵与应用仓库实际声明版本不一致时，以代码声明版本为准，并尽快修正文档。
